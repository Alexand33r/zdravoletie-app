# Ethical Considerations

## Data Classification Under GDPR

The dataset used in this study consists of Anovator body scan records collected from clients of Zdravoletie. Under the General Data Protection Regulation (EU) 2016/679, this data falls into two protected categories. First, the name field constitutes personal data under Article 4(1), as it directly identifies natural persons. Second, the biometric and health measurement fields — including body composition percentages, impedance values, postural risk scores, blood pressure, heart rate, vision acuity, and the derived bodyAge score — constitute special category data under Article 9(1), specifically data concerning health. Special category data carries heightened processing obligations and may only be processed under one of the explicit conditions listed in Article 9(2), the most applicable of which in a research context are explicit consent (Article 9(2)(a)) or processing for scientific research purposes with appropriate safeguards (Article 9(2)(j)).

Of the 158 records in the dataset, 103 carry the name value "Unknown", suggesting that the Anovator platform did not record or retain the client's identity for those scans. The remaining 50 records carry values that appear to be dates of birth encoded as six-digit strings (for example, "880224" for 24 February 1988), which are pseudonyms rather than true names but remain linkable to individuals within the clinic's operational records. No email addresses, national identification numbers, or contact details are present in the exported dataset. The original_url and id fields contain Anovator platform identifiers that could be used to retrieve additional personal data from the Anovator system by an operator with platform access; these fields were not used in any modelling or analysis and are excluded from the model feature set.

## Consent Assumptions

The data were collected in the course of normal clinical operations at Zdravoletie. Anovator scanning sessions are initiated by clinic staff as part of health assessment services to which clients have voluntarily subscribed. It is assumed that Zdravoletie's standard client intake process includes consent to collection and processing of biometric data for health assessment purposes. However, this study did not verify whether clients were specifically informed that their scan data would be used for machine learning research or shared with a student researcher. This represents a gap relative to the Article 9(2)(a) explicit consent standard for research use of special category data. If explicit research consent was not obtained at the point of data collection, processing under Article 9(2)(j) — scientific research with appropriate safeguards — would require a documented legitimate interest assessment and the application of the minimum necessary data principle, which in this context would include pseudonymisation of the name field and removal of platform identifiers before transfer to the researcher.

## Data Handling in This Study

The dataset was transferred to the researcher's local development environment as a CSV export and was not uploaded to any public repository, cloud storage service, or third-party platform during analysis. All modelling, experiment scripts, and visualisations were executed locally. The Streamlit dashboard does not transmit client data to external servers; all prediction and SHAP computation runs client-side within the Streamlit session. Model artifact files (.joblib) stored in the repository contain only fitted model parameters and no training records. No individual scan records are stored in the dashboard application or logged during use.

The git repository containing the code and model artifacts does not include the raw dataset file (Anovator_Biological_Master.csv). If the repository is published — for example, to accompany the thesis on an institutional repository — the data file must be excluded via .gitignore or equivalent mechanism, and the model artifacts should be reviewed to confirm they do not memorise or reconstruct individual records in a way that would constitute a data disclosure.

## Requirements for Production Deployment

A production deployment of the Streamlit dashboard or any successor system that processes real client scan data at the point of care would require the following before going live. A Data Protection Impact Assessment (DPIA) under Article 35 GDPR is mandatory for large-scale processing of health data, and the clinic would need to establish a lawful basis for each processing activity. Explicit informed consent forms covering research and analytical use of scan data would need to be integrated into the client onboarding process. If the dashboard is hosted on a cloud platform (such as Streamlit Community Cloud), a Data Processing Agreement under Article 28 GDPR must be in place with the hosting provider. Access to the dashboard should be restricted to authorised clinic staff through authentication, which is absent from the current research prototype. Audit logging of which client records were accessed and when would be required to satisfy Article 5(1)(f) data integrity and confidentiality obligations. Finally, a documented data retention policy specifying how long scan records are held and the procedure for responding to Article 17 erasure requests would be necessary before any live deployment.
